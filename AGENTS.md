# 项目规范（多角色朗读系列）

## 进入项目必读
**每次进入本项目时，必须先读取 `PROJECT_STATUS.md` 了解当前待办事项和任务状态。**
**如需详细历史记录，再读取 `TODO.md`。**
**每次进入本项目时，必须先读取 `朗读脚本代码规范.md` 了解制订阅读脚本的代码规范。**
**每次进入本项目时，第一项必须检查：所有输出（包括内部思考、工具调用说明、代码注释、文件内容）是否为全中文。**
**每次进入本项目时，必须先设置所有输出为中文，包括内部思考**
## 版本管理（硬性要求）
- **修改任何 `.json` 规则/插件文件前，必须先复制一份原文件作为备份**，新文件名按版本递增（如 `2.93` → `2.94`），禁止直接覆盖原文件。
- 新文件的以下字段必须同步更新：
  - 文件顶层 `name`
  - 文件顶层 `version`
  - `code` 内部的 `name:`
  - `code` 内部的 `version:`
- 版本号递增规则：沿用现有整数版本号（如 93 → 94），不要在代码内部使用与文件名不一致的版本号。
- **每次完成新版本后，必须执行 `git add . && git commit -m "版本说明" && git push origin master` 推送到远程仓库**（已配置一键双推，origin 会同时推送到 GitHub 和 cnb.cool 两个仓库），确保两边保持同步。
- **用户已一次性授权**：后续所有项目内操作（包括但不限于读取文件、修改文件、执行命令、`git push` 到远程仓库等）无需再次询问确认，由 AI 直接自动化执行。AI 仍会遵守系统级安全限制（如不访问工作目录外的敏感文件、不执行需要超级权限的操作等）。
- **每次修改后必须在 `TODO.md` 中记录变更内容**（版本号、改动点、新增/修复的功能），方便后续会话快速了解项目状态。
- **每次会话结束前，必须在 `TODO.md` 的「会话摘要」区块写入本次工作摘要**（当前版本、主目录结构、已完成事项、注意事项），确保下次会话能快速接上进度。

## 情绪模块扩展规范
- 如需融合外部插件（如猫剪豆问）的逻辑，优先以最小侵入方式扩展，保留原有开关和配置。
- 新增功能必须提供独立开关（如 `ENABLE_XXX`），且默认关闭，避免意外消耗 API 额度或破坏原有行为。
- 新增日志前缀需统一格式，如 `[AI]`、`[情绪桥接]`、`[情绪]` 等，便于用户识别来源。

## 代码风格
- 保持 ES5 兼容，不使用 ES6+ 语法（如 `let`、`const`、箭头函数、模板字符串等）。
- 新增变量/函数名避免与现有代码冲突，优先使用下划线前缀或模块化命名。

## JS 文件提取规范（方便 AI 调阅）
- 项目根目录下的 `.json` 规则/插件文件，其 `code` 字段（JavaScript 代码）必须同步提取到 `js/` 文件夹中，生成同名的 `.js` 文件。
- **每次修改 `.json` 文件后，必须运行 `node extract-js.js` 重新生成 `.js` 文件**，确保 `js/` 目录与 `.json` 保持同步。
- `js/` 下的 `.js` 文件**仅供人类阅读和 AI 调阅**，TTS Server 实际加载的是 `.json` 文件。
- `extract-js.js` 支持两种 JSON 结构：朗读规则（顶层 `.code`）和插件（子对象 `.code`）。
