# 项目规范（多角色朗读系列）

## 进入项目必读
**每次进入本项目时，必须先读取 `TODO.md` 了解当前待办事项和任务状态。**

## 版本管理（硬性要求）
- **修改任何 `.json` 规则/插件文件前，必须先复制一份原文件作为备份**，新文件名按版本递增（如 `2.93` → `2.94`），禁止直接覆盖原文件。
- 新文件的以下字段必须同步更新：
  - 文件顶层 `name`
  - 文件顶层 `version`
  - `code` 内部的 `name:`
  - `code` 内部的 `version:`
- 版本号递增规则：沿用现有整数版本号（如 93 → 94），不要在代码内部使用与文件名不一致的版本号。
- **每次完成新版本后，必须执行 `git add . && git commit -m "版本说明" && git push` 推送到远程仓库**，确保 GitHub 仓库与本地保持同步。

## 情绪模块扩展规范
- 如需融合外部插件（如猫剪豆问）的逻辑，优先以最小侵入方式扩展，保留原有开关和配置。
- 新增功能必须提供独立开关（如 `ENABLE_XXX`），且默认关闭，避免意外消耗 API 额度或破坏原有行为。
- 新增日志前缀需统一格式，如 `[AI]`、`[情绪桥接]`、`[情绪]` 等，便于用户识别来源。

## 代码风格
- 保持 ES5 兼容，不使用 ES6+ 语法（如 `let`、`const`、箭头函数、模板字符串等）。
- 新增变量/函数名避免与现有代码冲突，优先使用下划线前缀或模块化命名。
