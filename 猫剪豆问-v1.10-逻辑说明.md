# 猫剪豆问（自然情绪版）v1.0.2 逻辑说明

> 本文档介绍当前推荐使用的稳定版本：猫剪豆问朗读脚本（自然情绪版 v1.0.2）和配套 TTS 引擎（自然情绪版 v1.0.2）的核心逻辑与协作方式。

## 一、文件定位

| 文件 | 定位 | 作用 |
|------|------|------|
| `new/(脚本)猫剪豆问（自然情绪版）v1.0.2.json` | 朗读脚本 | 文本分析、角色识别、发音人标注、情绪桥接 |
| `new/猫剪豆问（自然情绪版）v1.0.2.json` | TTS 引擎 | 接收标注后的文本，调用猫箱 API 合成语音 |

两者是"前后道工序"关系：脚本先处理文本，引擎后合成语音。**v1.0.2 脚本必须与 v1.0.2 引擎配套使用**。

---

## 二、v1.0.2 脚本核心逻辑

### 1. 发音人角色配置

脚本内置三类发音人：

- **`MAIN_ROLES_CONFIG`**：主角色配置（当前为空，由外部列表决定）
- **`BATCH_ROLES`**：批量生成的通用发音人
  - 男主、女主：各 20 个
  - 少女/少年/女青年/男青年/女中年/男中年/女老年/男老年/女童/男童：各 300 个
- **`SPECIAL_ROLES`**：特殊场景发音人
  - `括号1`：【】括号内容
  - `括号2`：在线音效
  - `括号3`：「」括号内容
  - `括号4`：『对话旁白』

`FORCE_ENABLED_VOICE_TAGS` 强制启用 `括号1~4`，不受外部发音人列表过滤影响。

### 2. 角色启用过滤

脚本读取外部文件 `jiaoseliebiao-list.json`，根据其中 `speechRule.tag` 决定启用哪些发音人。如果读取失败，默认启用所有内置发音人。

启用后的发音人会写入 `fayinren.json`，按自然数顺序排序（如 `少女01`~`少女300`，而非字典序）。

### 3. 文本处理流程

每段文本进入脚本后，按以下顺序处理：

1. **异常引号拦截**：只处理"有右引号但无左引号"的特殊段落
2. **提取对话**：按中文双引号 `"..."` 提取对话
3. **无引号文本匹配**：没有对话时，尝试用章节缓存匹配多行无引号对话
4. **书籍切换检测**：有左引号时触发书籍切换处理
5. **章节缓存匹配**：优先用进度指针顺序命中，失败则全文定位
6. **AI 分析**：缓存未命中时调用大模型分析说话人
7. **发音人分配**：根据性别/年龄分配发音人，同一角色固定声音
8. **角色记录保存**：写入 `characterRecords.json`
9. **情绪桥接**：给对话和旁白添加 `[[emo:xxx]]` 前缀

### 4. 章节缓存与进度指针

- 以书籍名和章节标题为维度，将 AI 分析结果缓存到 `/Download/chajian/xiaoshuo/{书名}/{章节}.json`
- 新段落优先按进度指针顺序匹配缓存，命中失败则回退到全文定位
- 进度信息保存在 `reading_progress.json`，记录当前书籍、章节标题和最后处理到的对话序号

### 5. 异常引号与无引号文本处理

- **缺左引号**（如 `“你好呀！”张三说。`）：补全左引号，在引号后插入发音人标记，保留右引号后的旁白
- **缺右引号**：在左引号后插入标记，并在段落末尾补全右引号
- **完全无引号**：只在缓存中存在多行拆分对话且能匹配时，才补全双引号并标注发音人

### 6. 角色别名合并

AI 分析出新名字后，会再调用一次别名识别 AI，判断新名字是否为已有角色的别名（字号、绰号、昵称等）。如果是，则合并到已有角色的 `aliases` 字段；如果不是，则新增角色记录。

### 7. 情绪桥接逻辑

情绪桥接合并自原 obj2.js，给文本添加 `[[emo:xxx]]` 标记。

#### 对话情绪
- 本地关键词推断强情绪（愤怒、悲伤、紧张、惊讶、开心等）
- 无强情绪时继承上一段情绪（跨段落，5 分钟内有效）
- 在左引号后插入 `[[emo:xxx]]`
- 纯标点对话不推断、不继承、不写入缓存

#### 旁白情绪
- 本地关键词推断强情绪
- 无强情绪时继承上一段对话情绪
- 仍无情绪时使用默认兜底：`DEFAULT_NARRATION_EMOTION = "平静"`
- 在旁白开头插入 `[[emo:xxx]]`

#### 自然语言表演指令
- 根据情绪类型和场景温度（争吵/紧张/悲伤/温情）生成简单的自然语言表演指令
- 表演指令以 `[[emo:xxx|表演指令]]` 形式附加到情绪标记中
- 引擎会提取该指令并追加到 `context_texts` 中，影响 TTS 合成风格

## 三、v1.0.2 引擎核心逻辑

### 0. 顶部可调参数

引擎文件开头集中定义了一批可直接修改的配置项，无需改动核心逻辑：

| 参数名 | 默认值 | 说明 |
|--------|--------|------|
| `BASE_URL` | `wss://audio5-normal-hl.myparallelstory.com/internal/api/v1/ws` | 猫箱 TTS WebSocket 服务地址 |
| `AUDIO_FORMAT` | `mp3` | 输出音频格式 |
| `SAMPLE_RATE` | `24000` | 输出音频采样率 |
| `APP_KEY` | `WQuVLKMGRo` | 猫箱 API AppKey |
| `TIMEOUT_MS` | `30000` | 单次合成超时时间（毫秒） |
| `PITCH_VALUE` | `0` | 音高调整值 |
| `GLOBAL_SPEED_RATIO` | `1.7` | 全局语速倍率，最终速度 = `(speechRate / 20) * GLOBAL_SPEED_RATIO * 角色speed` |
| `GLOBAL_VOLUME` | `1.0` | 全局音量基准（1.0 为原声） |
| `SAVE_REQUEST_LOG` | `1` | 是否把每次合成参数写入日志（0 关闭，1 开启） |
| `SAVE_RESPONSE_LOG` | `1` | 是否记录每次合成结果（0 关闭，1 开启） |
| `LOG_RESET_PER_SESSION` | `1` | App 新会话首次合成时是否清空日志 |
| `LOG_MAX_SIZE_BYTES` | `1048576` | 日志文件大小上限（1MB），超过则清空 |
| `LOG_SESSION_KEY` | `tts_log_session_started_v1` | 内存缓存键，用于判断新会话 |
| `MAX_CHARS_PER_SYNTHESIS` | `500` | 单次合成最大字符数，超长文本自动切分 |
| `SEGMENT_RETRY_MAX` | `7` | 单段合成最大重试次数 |

> 修改方式：打开 `new/猫剪豆问（自然情绪版）v1.0.2.json`，找到引擎 `code` 字段开头对应变量，直接修改值即可。

### 1. 文本预处理（v1.0.2 关键改动）

引擎接收脚本处理后的文本，首先将特殊括号转换为带发音人标记的对话，**统一用中文双引号包裹**，确保 `<<tag>>` 能被引擎的对话分段逻辑正确消费：

- `【内容】` → `“<<括号1>>内容”`（排除纯数字序号如【1】【12】）
- `「内容」` → `“<<括号3>>内容”`
- `『内容』` → `“<<括号4>>内容”`

然后循环清洗对话内剩余的特殊括号，避免括号在已有对话内被读出。

### 2. 在线音效与本地音效

- 识别 `(中文音效)` 标记，下载或加载本地 JSON 音效文件
- 本地音效目录：`/storage/emulated/0/Download/chajian/bendiyinxiao/`
- 音效文件未找到时会从远程仓库下载并缓存
- 支持多音效轮播（`currentIndex` 循环）

### 3. 分段

引擎按中文双引号把文本切成对话段和旁白段：
- 引号内：对话
- 引号外：旁白

支持跨段对话：当对话左引号在上段、右引号在本段时，会把上段保存的 `pending voice` 应用到本段开头的孤儿右引号部分。

### 4. 发音人匹配

对话中的 `<<voiceTag>>` 被提取后，到 `tagConfig` 中查找对应配置。`tagConfig` 从 `jiaoseliebiao-list.json` 加载。

如果找不到对应配置，使用默认发音人 `zh_female_vv_uranus_bigtts`。

### 5. 情绪消费

- 提取 `[[emo:xxx]]` 或 `[[emo:xxx|表演指令]]` 标记
- 移除标记，避免被朗读
- 保存情绪值到 `cfg.emotion`
- 保存表演指令到 `cfg.performancePrompt`，追加到 `extraObj.context_texts`
- 不再要求对话/旁白 voice ID 含 `emo` 字样

### 6. 合成参数

合成音频时，只要 segment 配置中有 `emotion`，就传入：

```js
extraObj.audio_config.emotion = cfg.emotion;
extraObj.audio_config.emotion_scale = 4;
```

全局语速倍率 `GLOBAL_SPEED_RATIO = 1.7`，最终速度为：

```js
segRate = (speechRate / 20) * GLOBAL_SPEED_RATIO * cfg.speed;
```

### 7. 长文本切分

- `MAX_CHARS_PER_SYNTHESIS = 500`
- 超过 500 字的文本会按句号、感叹号、问号、逗号等标点自动切分为多个子段
- 避免单次合成过长导致音频截断或服务端返回 null

### 8. 合成重试与音频校验

针对 502/504/超时/空音频/音频过短等临时错误，引擎增强了自动重试：

- 最大重试次数：`SEGMENT_RETRY_MAX = 7`
- 重试间隔递增：500ms、900ms、1300ms、1700ms、2100ms、2500ms、2900ms
- 最小有效音频字节数：200 字节
- 7 次重试后仍失败：记录错误日志并**跳过本段**，继续合成后续段落

### 9. 合成日志

引擎顶部提供调试开关：

- `SAVE_REQUEST_LOG = 1`：把每次合成参数写入 `tts_request_log.txt`
- `SAVE_RESPONSE_LOG = 1`：记录合成结果
- 日志路径：`/storage/emulated/0/Download/chajian/mingwuyan/tts_request_log.txt`
- 新会话首次合成时自动清空日志（`LOG_RESET_PER_SESSION = 1`）
- 日志超过 1MB 时自动清空

日志中 `contextTexts`、`manualContextTexts`、`context_texts` 以及 `source.data` 中的对应字段会被替换为 `"[已省略设定]"`，实际发送给 TTS API 的请求仍包含完整设定。

## 四、脚本与引擎协作流程

```
原始文本
    ↓
v1.0.2 脚本
    ├── 异常引号/无引号文本处理
    ├── 识别对话/旁白
    ├── 章节缓存匹配 / AI 分析说话人
    ├── 分配发音人（<<voiceTag>>）
    └── 添加情绪标记（[[emo:xxx]]）
    ↓
标注后文本
    ↓
v1.0.2 引擎
    ├── 特殊括号转换为带引号对话
    ├── 提取 <<voiceTag>> 匹配发音人配置
    ├── 提取 [[emo:xxx]] 并移除标记
    ├── 长文本自动切分
    ├── 合成音频（带 emotion 参数）
    ├── 失败时自动重试 7 次，仍失败则跳过本段
    └── 可选：把合成参数写入 tts_request_log.txt
    ↓
音频输出
```

## 七、v1.0.2 关键修复细节

### 问题背景
v1.0.1 脚本在 main 逻辑中新增 `preprocessSpecialBrackets`，把 `「内容」` 等转换为 `<<括号3>>内容`，但**未用引号包裹**。引擎在分段时只有对话文本（带 `"..."`）才会消费 `<<tag>>` 发音人标记，旁白/普通文本中的 `<<括号3>>` 会被当成普通文本朗读，导致读出「括号3」。

### 修复方式
1. **脚本侧**：移除 v1.0.1 新增的 `preprocessSpecialBrackets` 函数及调用，恢复为 `var originalText = text`
2. **引擎侧**：移除原先会提前剥离所有括号的逻辑，改为在文本预处理阶段把特殊括号统一转换为带引号包裹的对话：
   - `【内容】` → `“<<括号1>>内容”`（排除纯数字序号如【1】【12】）
   - `「内容」` → `“<<括号3>>内容”`
   - `『内容』` → `“<<括号4>>内容”`
3. 保留对话内剩余括号清洗循环，避免括号在已有对话内被读出
4. 纯数字序号如 `【1】`、`【12】` 不受影响，继续作为角色序号处理
5. `括号2`（在线音效）仍保持原有逻辑

## 八、注意事项

1. **v1.0.2 脚本需与 v1.0.2 引擎配套使用**，混用 v1.0.1 或更早版本可能导致括号标记被读出或情绪丢失。
2. 该版本基于 v1.0，**未引入 v1.1+ 的完整表演链路**，避免 OOM 风险。
3. 情绪桥接默认开启，如需关闭可修改脚本中 `ENABLE_EMOTION_BRIDGE` 的值。
4. 跨段对话依赖内存缓存 `maoxiang_pending_voice_v1`，App 进程重启后会丢失，下一段开头会重新分析。
5. 长文本切分以 500 字为阈值，切分后的子段会复用同一份发音人/情绪配置。
