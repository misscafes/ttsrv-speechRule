# 朗读脚本代码规范（Rhino JS 引擎兼容指南）

> 本文档基于 `朗读规则2.js`、`并发脚本.js`、`正常脚本.js` 的开发/调试经验整理，适用于 Legado 阅读 App 中的「朗读脚本」功能。
> 
> 朗读脚本在 **Mozilla Rhino 1.8.1** JS 引擎中执行，虽然源码设置了 `VERSION_ES6`，但该版本 ES6 实现**残缺且有 bug**，实际应按 **ES5.1 + Rhino 兼容补丁** 的标准编写。

| 项目 | 说明 |
|------|------|
| JS 引擎 | Mozilla Rhino 1.8.1（2016 年版本） |

## 三、必须遵守的规则（否则直接报错）

### 3.1 【致命】顶层代码禁止使用 `return`

Rhino 的 `evaluateReader` 执行的是「脚本」而非「函数」，**脚本顶层不允许出现 `return`**。

```javascript
// ❌ 错误：直接 eval 时，顶层 return 会抛出 "invalid return"
var result = doSomething();
return result;

// ✅ 正确：用 IIFE 包装，return 变成函数内部返回
(function() {
    var result = doSomething();
    return result;
})()
```

> **原因**：朗读脚本直接被 App `eval(script.code)`，没有外层函数包裹，因此所有主逻辑必须包在 IIFE 中。

---

### 3.2 【致命】禁止使用 ES6 语法和方法

Rhino 1.8.1 虽然声明支持 ES6，但以下方法/语法**在 regex、字符串处理、对象操作等场景下会抛出异常或表现异常**：

| 禁止项 | 错误示例 | 替代方案 |
|--------|---------|---------|
| `String.prototype.endsWith` | `"abc".endsWith("c")` | `str.lastIndexOf("c") === str.length - 1` |
| `String.prototype.startsWith` | `"abc".startsWith("a")` | `str.indexOf("a") === 0` |
| `String.prototype.includes` | `"abc".includes("b")` | `str.indexOf("b") !== -1` |
| `Array.prototype.map` | `arr.map(fn)` | 手写 `for` 循环构建新数组 |
| `Array.prototype.filter` | `arr.filter(fn)` | 手写 `for` 循环 + `if` 判断 |
| `let` / `const` | `let x = 1` | 统一用 `var` |
| 箭头函数 | `x => x + 1` | `function(x){ return x + 1; }` |
| 模板字符串 | `` `Hello ${name}` `` | `"Hello " + name` |
| 解构赋值 | `var {a, b} = obj` | `var a = obj.a, b = obj.b` |
| 默认参数 | `function(a, b=1){}` | `function(a, b){ b = b || 1; }` |
| `Object.assign` | `Object.assign({}, a)` | 手写 `for...in` 拷贝 |
| `Array.isArray` | `Array.isArray(x)` | `Object.prototype.toString.call(x) === "[object Array]"` |

### 3.3 【致命】`java.readExternalFile()` 返回的是 Java String，必须显式转换

`java.readExternalFile()` 返回的是 Java 的 `java.lang.String` 对象，在 Rhino 中表现为 `NativeJavaObject`，**不是 JS 原生 String**。

这会导致以下隐蔽 bug：

```javascript
// ❌ 错误：Java String 和 JS String 类型不同，=== 永远为 false
var content = java.readExternalFile("/path/to/file");
if (content === "") {  // 永远进不来！
    // ...
}

// ❌ 错误：Java String.split() 返回 Java String[]，没有 .map() 方法
var arr = content.split("@@").map(function(s){ return s.trim(); });
// 抛出 TypeError: arr.map is not a function

// ✅ 正确：第一行就强制转为 JS String
var content = String(java.readExternalFile("/path/to/file"));
if (content === "") {  // 正常工作
    // ...
}
// ✅ 正确：split 后手动 for 循环处理
var parts = content.split("@@");
for (var i = 0; i < parts.length; i++) {
    parts[i] = String(parts[i]).trim();
}
```

**所有从 `java.readExternalFile()` 读取的内容，第一行必须 `String(content)`。**

同理，`java.ajax()` 返回的也是 Java String，需要 `String(response)` 转换。

### 3.4 【致命】正则中禁止使用 `\uXXXX` Unicode 转义（仅限 `new RegExp()`）

Rhino 1.8.1 的 `RegExp` 构造函数**不会解析字符串中的 `\uXXXX` 转义序列**。

```javascript
// ❌ 错误：new RegExp 不会把 \u201c 解析成中文左双引号
var reg = new RegExp("[\u201c\u201d]", "g");
// 实际上匹配的是字面的 "u201c" 和 "u201d" 这 5 个字符

// ✅ 正确：用 String.fromCharCode() 生成真实字符
var LQ = String.fromCharCode(0x201C); // 真正的 "
var RQ = String.fromCharCode(0x201D); // 真正的 "
var reg = new RegExp("[" + LQ + RQ + "]", "g"); // 正常工作

// ⚠️ 注意：正则字面量 /.../ 中的 \uXXXX 是 JS 解析器处理的，可以工作
var reg2 = /[\u4e00-\u9fa5]/g;  // 这个在 Rhino 里通常可以
// 但为了绝对安全，建议所有 Unicode 字符都用 String.fromCharCode()
```

**推荐做法**：文件开头集中定义所有 Unicode 字符变量，全文不再写 `\uXXXX`：

```javascript
var LQ = String.fromCharCode(0x201C); // "
var RQ = String.fromCharCode(0x201D); // "
var CJK_S = String.fromCharCode(0x4E00); // 汉字起始
var CJK_E = String.fromCharCode(0x9FA5); // 汉字结束
// ... 等等
```

### 3.5 【致命】禁止访问被 RhinoClassShutter 屏蔽的 Java 类

以下类在朗读脚本中**不可访问**（会抛出安全异常）：

- `java.io.File` / `FileInputStream` / `FileOutputStream`
- `java.nio.file.Paths` / `Files`
- `java.lang.Runtime` / `ProcessBuilder`
- `java.lang.ClassLoader` / `Class`
- `java.lang.reflect.*`
- `android.content.Context` / `Intent`
- `dalvik.system.*`
- `cn.hutool.core.io.*`

**文件操作必须使用 `java` 对象提供的方法**：

```javascript
// ✅ 正确：通过 java 扩展对象读写外部文件（限制在 /storage/emulated/0/Download/ 目录）
var content = String(java.readExternalFile("/storage/emulated/0/Download/test.txt"));
java.writeExternalFile("/storage/emulated/0/Download/out.txt", "hello");

// ❌ 错误：直接 new File
var file = new java.io.File("/path/to/file"); // 安全异常
```

## 四、返回值与执行规则

### 4.1 返回值规范

脚本最终需要返回一个字符串，作为修改后的朗读文本。有以下两种方式：

1. **直接写 `text` 变量作为最后一行**（推荐）
   ```javascript
   text = text.replace(/某个词/g, "替换词");
   text  // 最后一行表达式即为返回值
   ```

2. **用 IIFE 显式返回**（复杂脚本推荐）
   ```javascript
   (function() {
       var processed = text.replace(/某个词/g, "替换词");
       return processed;
   })()
   ```

### 4.2 执行规则

1. **多个脚本**：按列表中从上到下顺序执行，上一个脚本的输出作为下一个脚本的输入
2. **禁用脚本**：未启用的脚本不会执行
3. **错误处理**：单个脚本执行出错时，会跳过该脚本并记录日志，不影响后续脚本和朗读流程
4. **空脚本**：代码为空的脚本会被跳过

### 5.3 数组操作规范

Rhino 中**不要假设任何 ES5+ 的 Array 方法存在**，全部手写：

```javascript
// ❌ 禁止
var filtered = arr.filter(function(x){ return x > 0; });

// ✅ 推荐
var filtered = [];
for (var i = 0; i < arr.length; i++) {
    if (arr[i] > 0) filtered.push(arr[i]);

var mapped = [];
for (var i = 0; i < arr.length; i++) {
    mapped.push(arr[i] * 2);
}
```

## 七、阶梯式并发请求详解（重点）

### 7.1 为什么需要阶梯式并发？

朗读脚本在每个段落合成前都会执行。如果脚本内部需要调用 AI API（如角色分析、文本改写等），**串行调用存在以下问题**：

1. **单点故障**：一个 API 超时或失败，整段朗读卡住或降级
2. **超时无法补救**：串行模式下第1个请求超时后，第2个请求才开始发，第1个请求白白浪费
3. **用户体验差**：长文本朗读时，每段都等待 API 返回，导致朗读不连贯

**传统并发（`ajaxRace`）的问题**：
- 所有请求**同时**启动，如果第1个请求本来就能成功，第2、3个请求就是浪费
- 对单 API 场景无效（只配1个密钥时，并发和串行没区别）

**阶梯式并发（`ajaxRaceDelayed`）的优势**：
- 请求**逐个延迟启动**，已启动的请求继续保留运行
- 第1个请求启动后，等固定间隔时间到了才启动第2个，第1个继续跑不受影响
- 即使只配了**1个API**，也能实现**超时后自动补发**的效果
- 总耗时 ≈ 最快成功的那个请求耗时，不会浪费不必要的请求

### 7.2 阶梯式并发方案：`java.ajaxRaceDelayed()`

Rhino 1.8.1 **不支持原生多线程**（没有 `Promise`、`async/await`、`Worker`），无法在 JS 层实现真正的并发。

解决方案：**利用 Kotlin/Java 层提供并发能力**，在 App 层封装一个阶梯式并发竞速方法，暴露给 JS 调用。

```javascript
// ✅ 阶梯式并发竞速：逐个延迟启动，保留已启动的请求继续运行
var raceBody = java.ajaxRaceDelayed(requestList, RACE_DELAY);
```

**核心原理**：
- JS 层负责构造多个请求配置（JSON 字符串数组）
- 通过 `java.ajaxRaceDelayed()` 将请求列表和延迟间隔传递到 Kotlin 层
- Kotlin 层使用原生 `Thread + CountDownLatch` 实现并发竞速，**不使用协程**（避免在协程内部嵌套 `runBlocking` 导致死锁）
- 已启动的请求在后台线程继续运行，不被取消
- 哪个请求先成功返回 200，就把它的 body 回传给 JS
- 其他未完成的请求通过 `thread.interrupt()` 中断取消

**时序行为**（假设 `RACE_DELAY = 8000`）：

```
t=0s    启动请求1
        ↓
t=8s    启动请求2（请求1继续跑）
        ↓
t=16s   启动请求3（请求1/2继续跑）
        ↓
任意时刻  请求2先成功返回 → 立刻结束，请求1/3自动取消
```

**关键设计**：
- `delayMs`（第2个参数）是**相邻请求的启动间隔**，与底层请求超时无关
- 底层请求已设置 `ignoreTimeout = true`，单个请求最长可跑 5 分钟，不会因超时断开
- 第1个请求启动后，如果它在 `RACE_DELAY` 内成功返回，后面的请求根本不会启动
- 如果第1个请求响应慢，第2个请求会在固定间隔后补位启动，两个请求同时竞争

### 7.3 脚本配置变量说明

```javascript
var ANALYZE_TIMEOUT = 25000;  // 保留兼容，请求体中的 timeout 字段已弃用，底层统一忽略
var RACE_DELAY = 8000;        // 阶梯间隔：相邻请求的启动间隔（毫秒）
var MAX_RETRY = 3;            // 阶梯并发总请求数（1个配置会重复发3次）
```

| 变量 | 作用 | 建议值 |
|------|------|--------|
| `ANALYZE_TIMEOUT` | 保留兼容的历史变量，请求体中已注释掉 `timeout` 字段，底层统一忽略超时 | 20000~30000 |
| `RACE_DELAY` | `ajaxRaceDelayed` 的间隔参数，固定间隔启动备用请求 | 5000~10000 |
| `MAX_RETRY` | 阶梯并发的总请求数 | 2~4 |

**两者配合的效果**：
- `RACE_DELAY = 8000`，`MAX_RETRY = 3`
- 最多同时有 3 个请求在跑，相邻启动间隔 8 秒
- 最坏情况总耗时 ≈ 8s × 2 + API 响应时间（第3个请求启动后需要等它返回）

### 7.4 并发脚本完整示例

var MAX_RETRY = 3;             // 阶梯并发总请求数
var BELOW_EXTRACT_LENGTH = 800;
var MAX_HISTORY = 20;
var HISTORY_FILE = EXT_DIR + "paragraphHistory.json";

    try { java.log(msg); } catch (e) {}
}

function padZero(num, length) {
    num = num.toString();
    while (num.length < length) num = "0" + num;
    return num;
}

// ===================== 密钥管理（支持多密钥）=====================
function readApiConfigs() {
    try {
        var content = String(java.readExternalFile(KEY_FILE));
        if (!content || content.trim() === "") return [];
        var contentTrim = content.trim();
        // 有##分隔：只取第一部分

            contentTrim = contentTrim.split("##")[0].trim();
        }
        if (contentTrim === "") return [];
        // 有@@：按 地址@@模型@@密钥 分组
        if (contentTrim.indexOf("@@") !== -1) {

            for (var i = 0; i + 2 < splitArr.length; i += 3) {

        return [];
    }

// ===================== 阶梯式并发调用 AI =====================
function analyzeWithRaceDelayed(seqContent, aboveContext) {
    var userContent = "";
    if (aboveContext && aboveContext.trim() !== "") {
        userContent += "【上文历史内容】\n" + aboveContext + "\n";
    }
    userContent += "【当前待分析对话内容】\n" + seqContent;

    // 构建多个 API 配置的请求数组
    // 按 MAX_RETRY 次数构建，配置不足时轮询复用
    // 即使只配了1个API，也会重复发MAX_RETRY次，实现超时后继续发送的效果
    var requestList = [];
    var configs = readApiConfigs();
    if (configs.length === 0) configs = [DEFAULT_API_CONFIG];

    for (var i = 0; i < MAX_RETRY; i++) {
        var cfg = configs[i % configs.length];   // 轮询复用配置
        var requestBody = JSON.stringify({
            model: cfg.model,   // 必须用 cfg.model，不能用 DEFAULT_API_CONFIG.model
            messages: [
                { role: "system", content: "你的系统提示词..." },
                { role: "user", content: userContent }
            ],
            temperature: 0.1
        });
        requestList.push(JSON.stringify({
            url: cfg.endpoint,
            method: "POST",
            body: requestBody,
            headers: {
                "Content-Type": "application/json",
                "Authorization": "Bearer " + cfg.key
            }
            // timeout: 已弃用；ajaxRaceDelayed 内部忽略底层超时，请求会一直活着直到外层 cancel
        }));
    }

    log("【AI】阶梯并发 " + requestList.length + " 个请求，间隔 " + RACE_DELAY + "ms...");

    // ---------- 阶梯式并发核心 ----------
    // 第1个请求立即启动，8秒后启动第2个，16秒后启动第3个...
    // 谁先成功返回谁的结果，其余自动取消
    var raceBody = java.ajaxRaceDelayed(requestList, RACE_DELAY);
    // ------------------------------------

    if (raceBody) {
        try {
            var resJson = JSON.parse(raceBody);
            if (resJson.choices && resJson.choices[0] && resJson.choices[0].message) {
                var content = resJson.choices[0].message.content.trim();
                var jsonStart = content.indexOf("{");
                var jsonEnd = content.lastIndexOf("}");
                if (jsonStart !== -1 && jsonEnd !== -1) {
                    var result = JSON.parse(content.substring(jsonStart, jsonEnd + 1));
                    // 校验字段完整性
                    var valid = true;
                    for (var key in result) {
                        if (result.hasOwnProperty(key)) {
                            var item = result[key];
                            if (!item || !item.name || !item.gender || !item.age) {
                                valid = false;
                                break;
                            }
                        }
                    }
                    if (valid) {
                        log("【AI】阶梯并发成功，" + Object.keys(result).length + " 个结果");
                        return result;
                    }
                }
            }
        } catch (e) {
            log("【AI】解析阶梯并发结果失败：" + e.message);
        }
    }
    return null;
}

### 7.5 多密钥配置格式

在 `miyue.txt` 中配置 API 密钥，支持三种格式：

**格式一：多组完整配置（推荐）**
```
https://api1.com@@model1@@key1@@https://api2.com@@model2@@key2
```
每组为 `地址@@模型@@密钥`，多组之间用 `@@` 连接。

**格式二：纯密钥**
```
your-api-key-here
```
只填密钥时使用默认 endpoint 和 model。

**格式三：混合分隔（有##时只取前半部分）**
```
地址@@模型@@密钥##其他内容
```
`##` 后面的内容会被忽略，方便和其他配置共存。

### 7.6 不同配置数量下的并发行为

| 配置数 | MAX_RETRY | requestList | 实际行为 |
|--------|-----------|-------------|----------|
| 1 | 3 | 3个相同的请求 | t=0s发第1个，t=8s发第2个，t=16s发第3个 |
| 2 | 3 | 3个请求（API1、API2、API1） | t=0s发API1，t=8s发API2，t=16s再发API1 |
| 3 | 3 | 3个请求（API1、API2、API3） | t=0s发API1，t=8s发API2，t=16s发API3 |
| 1 | 2 | 2个相同的请求 | t=0s发第1个，t=8s发第2个 |

**关键点**：
- 只配1个API时，通过 `MAX_RETRY` 实现**同一API的超时补发**
- 多API时，通过 `configs[i % configs.length]` 实现**轮询 + 超时补发**

### 7.7 阶梯式并发 vs 串行对比

| 特性 | 串行（正常脚本.js） | 阶梯式并发（并发脚本.js） |
|------|-------------------|------------------------|
| 核心方法 | `org.jsoup.Jsoup.connect()` | `java.ajaxRaceDelayed()` |
| 单API超时处理 | 超时后才开始第2次请求 | 超时后补发新请求，旧请求继续跑 |
| 多API利用 | 轮询逐个试 | 阶梯启动，取最快 |
| 已超时请求 | 浪费（已断开） | 保留继续运行（可能慢但还能返回） |
| 实现复杂度 | 简单 | 需 App 层支持 `ajaxRaceDelayed` |
| 适用场景 | 单密钥、低延迟 API | 单/多密钥、高可用需求 |
| 总请求数控制 | `while` 循环计数 | `MAX_RETRY` 固定上限 |

### 7.8 并发使用注意事项

1. **`ajaxRaceDelayed` 需要 App 层支持**：不是标准 Rhino/Legado 功能，需要修改 App 源码添加 `JsExtensions.ajaxRaceDelayed()` 方法
2. **`delayMs` 参数**：第2个参数是相邻请求的启动间隔，使用固定的 `RACE_DELAY`，与请求超时无关
3. **请求数量控制**：`MAX_RETRY` 不要设太大（建议 2~4），避免过多请求堆积
4. **结果校验**：并发返回的结果和串行没有区别，仍需做 JSON 校验和字段完整性检查
5. **日志记录**：建议打印 `requestList.length` 和 `RACE_DELAY`，方便排查时序问题
6. **`model` 字段**：每个请求的 `model` 必须用 `cfg.model`，不能用 `DEFAULT_API_CONFIG.model`，否则多API配置会失效

## 八、缓存机制详解

### 8.1 内存缓存（`cache` 对象）

- **生命周期**：App 进程存活期间有效，杀后台后清空
- **适用场景**：临时保存 API 规则、配置信息，避免每次朗读都读文件
- **限制**：只能存字符串，复杂对象需 `JSON.stringify`

```javascript

if (!rulesStr) {
    var resp = java.ajax(API_URL, 5000);
    cache.put("tts_replace_rules", resp);  // 内存缓存
    rulesStr = resp;
}
```

### 8.2 文件缓存（`readExternalFile/writeExternalFile`）

- **生命周期**：持久化存储，App 重启后仍在
- **适用场景**：AI 分析结果、对话角色缓存、上文历史记录
- **限制**：只能读写 `/storage/emulated/0/Download/` 目录下文件

## 九、上文/下文机制

### 9.1 上文历史（Paragraph History）

记录最近朗读的 N 个段落文本，为 AI 提供上下文语境。

```javascript
var MAX_HISTORY = 20;
var HISTORY_FILE = EXT_DIR + "paragraphHistory.json";

function readParagraphHistory() {
    try {
        var content = String(java.readExternalFile(HISTORY_FILE));
        if (!content || content.trim() === "") return [];
        var arr = JSON.parse(content);
        return Object.prototype.toString.call(arr) === "[object Array]" ? arr : [];
    } catch (e) { return []; }
}

function saveParagraphHistory(para) {
    try {
        if (!para || para.trim() === "") return;
        var arr = readParagraphHistory();
        arr.push(para);
        if (arr.length > MAX_HISTORY) arr.shift();  // 队列溢出时移除最旧的
        java.writeExternalFile(HISTORY_FILE, JSON.stringify(arr));
    } catch (e) {}
}
```

### 9.2 下文提取（Below Content）

从整章文本中提取当前段落之后的部分内容，帮助 AI 预判后续对话。

```javascript
function getBelowContent(currentParagraph) {
    try {
        var dataStr = String(java.readExternalFile(DATA_FILE));
        if (!dataStr || dataStr.trim() === "") return "";
        var data = JSON.parse(dataStr);
        if (!data || !data.texts) return "";
        var fullChapter = String(data.texts);

        // 先完整匹配，再降级为前50字短匹配
        var targetPos = fullChapter.indexOf(currentParagraph);
        if (targetPos === -1) {
            var shortPara = currentParagraph.substring(0, Math.min(currentParagraph.length, 50));
            targetPos = fullChapter.indexOf(shortPara);
            if (targetPos === -1) return "";
        }

        var startExtractPos = targetPos + currentParagraph.length;
        var remainLen = fullChapter.length - startExtractPos;
        if (remainLen <= 0) return "";
        return fullChapter.substring(startExtractPos, startExtractPos + Math.min(BELOW_EXTRACT_LENGTH, remainLen));
    } catch (e) { return ""; }
}
```

**实现要点**：
1. `data.json` 需要提前准备好，包含完整章节文本（`texts` 字段）
2. 匹配失败时降级为短文本匹配，提高容错
3. 下文长度建议 500~800 字，太长增加 AI 分析耗时，太短缺少语境

## 十二、常见错误对照表

| 错误现象 | 可能原因 | 解决方案 |
|---------|---------|---------|
| `invalid return` | 顶层代码写了 `return` | 包进 IIFE |
| `TypeError: xxx.map is not a function` | `java.readExternalFile().split()` 返回 Java Array | `String()` 转换后再 split |
| `TypeError: xxx.endsWith is not a function` | 用了 ES6 方法 | 改用 `indexOf` / `lastIndexOf` |
| 空文件判断失效（明明没内容却往下走）| `content === ""` 因类型不同为 false | `String(content)` 后再判断 |
| 正则匹配不到中文双引号 | `new RegExp("[\u201c]")` 没解析 | `String.fromCharCode(0x201C)` |
| `SecurityException: 非法路径` | `writeExternalFile` 路径不在 Download 下 | 确保路径以 `/storage/emulated/0/Download/` 开头 |
| `ClassShutter: prohibited package/name` | 访问了被屏蔽的 Java 类 | 改用 `java` 对象提供的方法 |
| `ajaxRaceDelayed is not a function` | App 层未实现阶梯式并发方法 | 使用串行版本或修改 App 源码 |
| 并发请求全部失败 | 密钥错误或网络问题 | 检查 `miyue.txt` 格式和密钥有效性 |
| AI 分析结果错位 | Prompt 中序号标记与返回格式不匹配 | 检查 `【01】` 标记和返回 JSON 的 key 一致性 |

---

> **最后提醒**：Rhino 1.8.1 是一个相当古老的引擎，不要用现代浏览器 JS 的思维去写。最安全的写法是：**ES3 语法 + 手写循环 + 显式类型转换**。阶梯式并发请求能显著提升单/多密钥场景下的稳定性（超时后自动补发，已超时的请求继续保留运行），但需要 App 层的配合支持，使用前务必确认 `java.ajaxRaceDelayed` 可用。
